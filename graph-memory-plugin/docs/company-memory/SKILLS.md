# Shared Skills — Company-Wide Skill Architecture

> How organizational procedures become executable agent skills.

## Concept

Currently, Skillforge generates skills from individual high-access nodes. For company memory, we extend this: company-wide procedures, patterns, and decisions become skills available to every employee's agent.

## Two Skill Sources

```
Personal Skills (~/.agents/skills/, ~/.claude/skills/)
├── Personal patterns discovered by individual
├── Generated by personal Skillforge
└── Private to individual

Company Skills (~/.graph-memory/company/skills/)
├── Organizational procedures and patterns
├── Generated by company Skillforge or hand-authored
└── Inherited by all employees
```

Both sources are discovered by the agent harness. Company skills are namespaced `[company]` in listings.

## Company Skill Categories

### Procedure Skills
Executable step-by-step procedures that the company has standardized:

```
skills/
├── deploy-to-production/SKILL.md      # "How we ship to Railway"
├── incident-response/SKILL.md         # "Our incident procedure"
├── code-review/SKILL.md               # "Our review standards"
├── database-migration/SKILL.md        # "How we handle schema changes"
├── on-call-handoff/SKILL.md           # "Shift handoff procedure"
├── feature-flag/SKILL.md              # "How we use feature flags"
└── hotfix/SKILL.md                    # "Emergency hotfix procedure"
```

### Pattern Skills
Common patterns that every engineer should know:

```
skills/
├── error-handling/SKILL.md            # "Our error handling patterns"
├── testing-standards/SKILL.md         # "What we test and how"
├── api-design/SKILL.md                # "Our API design conventions"
└── logging/SKILL.md                   # "What we log and how"
```

### Onboarding Skills
New-hire-specific procedures:

```
skills/
├── first-day-setup/SKILL.md           # "Set up your dev environment"
├── first-deploy/SKILL.md              # "Your first production deploy"
└── first-pr/SKILL.md                  # "Your first code review"
```

## Skill Generation Pipeline

### Company Skillforge

The company daemon runs Skillforge on company nodes, similar to the personal Skillforge but with procedure-oriented prompts:

```
Company nodes (decisions, procedures, patterns)
         │
         ▼
    ┌──────────┐
    │ Skillforge │ ← Procedure-oriented prompt template
    │ (company)  │
    └──────┬───┘
           │
           ▼
    skills/.drafts/{name}/SKILL.md
           │
           ▼
    ┌──────────┐
    │  Review   │ ← Domain steward approves
    └──────┬───┘
           │
           ▼
    skills/{name}/SKILL.md  (promoted to production)
```

### Skill Manifest

Each company skill includes provenance metadata:

```yaml
# skills/deploy-to-production/SKILL.md
---
name: deploy-to-production
description: "Deploy a service to Railway production"
provenance: auto-generated | human-authored
source_nodes:
  - procedures/deploy-to-production
  - decisions/railway-three-environments
author: company-skillforge  # or individual GitHub username
team: platform
reviewed_by: patrick  # steward who approved
reviewed_at: 2026-05-29
version: 2
---
```

## Skill Discovery

The agent harness discovers skills from multiple paths:

```typescript
interface SkillDiscovery {
  // Existing paths (personal)
  personalPaths: [
    '~/.agents/skills/',
    '~/.claude/skills/',
    '~/.config/opencode/skill/',
  ]
  
  // New path (company)
  companyPath: '~/.graph-memory/company/skills/'
}
```

The plugin reads skills from all paths. Company skills are displayed with a `[company]` prefix in skill listings:

```
Available skills:
  [company] deploy-to-production — Deploy a service to Railway production
  [company] incident-response — Our incident procedure
  [company] code-review — Our review standards
  adapt — Adapt designs for responsive layouts
  animate — Add animations and micro-interactions
  ...
```

## Skill Sync

Company skills are synced via the same git pull mechanism as other company data:

```
Individual daemon tick:
1. git pull --rebase (company repo)
2. Diff skills/ directory against cached state
3. If new/updated skills:
   a. Read skill manifests
   b. Update local skill registry
   c. Log: "2 new company skills available: hotfix, feature-flag"
```

No special sync mechanism — skills are just files in the company repo.

## Hand-Authored vs. Auto-Generated

### Hand-Authored Skills
- Written by engineers or team leads
- Committed directly to `company-graph/skills/{name}/SKILL.md`
- Higher quality, intentional
- Good for: procedures, standards, onboarding

### Auto-Generated Skills
- Created by company Skillforge from node patterns
- Written to `skills/.drafts/{name}/SKILL.md` first
- Require steward review before promotion
- Good for: emerging patterns, discovered best practices

### Coexistence

```yaml
# Company skillforge config
skillforge:
  generation_mode: suggest  # suggest | auto
  # suggest: writes to .drafts/, requires review
  # auto: writes directly to skills/ (risky, for mature orgs only)
  
  categories:
    - procedures  # Always auto-generate from procedure nodes
    - patterns    # Auto-generate from high-confidence patterns
    # Don't auto-generate from: decisions, incidents, people
```

## Skill Health

Company daemon tracks skill health:

```typescript
interface SkillHealth {
  name: string
  last_updated: Date
  source_nodes_valid: boolean  // Are referenced nodes still active?
  access_count: number         // How often is this skill loaded?
  staleness_score: number      // 0-1, higher = more stale
}
```

- Skills whose source nodes are archived → flagged for review
- Skills not accessed in 90 days → flagged for archival
- Skills with stale content (source node updated but skill not regenerated) → auto-regenerate
